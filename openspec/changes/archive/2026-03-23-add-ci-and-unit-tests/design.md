# Design: Testing Architecture & CI Workflows

## 1. Testing Architecture (Hardware Decoupling)
To make the application testable on a standard CI runner without a Bluetooth chip, we will introduce a Dependency Injection (DI) friendly approach (even if done manually without Dagger/Hilt to keep it lightweight).

- **`IBluetoothSender` (Interface):** Defines methods like `sendReport(reportId: Int, data: ByteArray)`.
- **`BluetoothHidManager` (Implementation):** The actual Android hardware implementation using `BluetoothHidDevice`. This class will *not* be unit tested in CI.
- **`MockBluetoothSender` (Test Implementation):** Used in unit tests to capture and assert the `ByteArray` generated by the input interceptors.

## 2. Target Test Coverage
- **`AzertyToQwertyMapperTest`:** Assert that `KeyEvent.KEYCODE_A` outputs the HID usage code for 'Q', and `KEYCODE_Z` outputs 'W', etc.
- **`HidReportConfigTest`:** Assert that the generated Composite HID Descriptor byte array has the exact expected size and values.

## 3. GitHub Actions Workflows
Located in `.github/workflows/`:

### A. `build-apk.yml` (Automated)
- **Trigger:** `on: [push, pull_request]`
- **Runner:** `ubuntu-latest`
- **Steps:**
  1. Checkout repository.
  2. Setup JDK 17.
  3. Setup Gradle caching.
  4. Run `./gradlew assembleDebug`.
  5. Use `actions/upload-artifact` to expose the generated `app-debug.apk`.

### B. `manual-unit-tests.yml` (Manual)
- **Trigger:** `on: [workflow_dispatch]`
- **Runner:** `ubuntu-latest`
- **Steps:**
  1. Checkout repository.
  2. Setup JDK 17.
  3. Setup Gradle caching.
  4. Run `./gradlew testDebugUnitTest` to execute local JVM tests.
